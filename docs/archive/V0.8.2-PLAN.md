## Technical Architecture Decisions

### 1. Source Provider Trait Design

**Decision:** Implement the `SourceProvider` + `ContentLoader` two-phase model from V82DEEPDIVE.md

**Rationale:**
- Enumeration is cheap (listing files/entries)
- Content loading is expensive (reading blobs)
- Separating them allows skip decisions *before* loading content
- Critical for Docker images with 50K+ files

**Implementation:**

```rust
// crates/coax-scanner/src/source_provider.rs

/// A scannable unit's metadata (without content)
pub struct ScanTarget {
    pub origin: ScanOrigin,
    pub size_hint: Option<u64>,
    pub content_type: Option<ContentType>,
}

/// Where content came from
pub enum ScanOrigin {
    FileSystem { path: PathBuf },
    GitBlob { commit: String, path: PathBuf, author: String, date: DateTime<Utc> },
    GitDiff { commit: String, path: PathBuf, added: bool },
}

/// The source provider trait
pub trait SourceProvider: Send + Sync {
    /// Enumerate what's available (cheap, no content loading)
    fn enumerate(&self) -> Box<dyn Iterator<Item = ScanTarget> + '_>;
    
    /// Total expected units for progress reporting
    fn total_units_hint(&self) -> Option<u64>;
    
    /// Human-readable unit name for progress bar
    fn progress_unit_name(&self) -> &str;
}

/// Content loading (expensive, called selectively)
pub trait ContentLoader: Send + Sync {
    fn load(&self, target: &ScanTarget) -> Result<ScanContent>;
}

pub enum ScanContent {
    Buffered(Vec<u8>),
    Streamed(Box<dyn Read>),
}
```

**Potential Issues:**
- Lifetime management with git2 objects (may need `Arc<Repository>`)
- Streaming large blobs through the scanner (scanner currently expects `&str`)

### 2. Git History Implementation

**Decision:** Use `git2` crate (libgit2 bindings), not shelling out to `git` CLI

**Rationale:**
- Faster (no process spawn overhead per commit)
- No git binary dependency
- Structured access to objects
- Better error handling

**Key Design Choices:**

#### 2.1 Merge Commit Handling
**Decision:** Diff against **first parent only** for merge commits

**Rationale:** Prevents double-counting secrets that were already reported on the feature branch

```rust
let parent = commit.parent(0)?;  // First parent only
```

#### 2.2 Shallow Clone Detection
**Decision:** Detect and warn explicitly

**Implementation:**
```rust
if repo.is_shallow() {
    eprintln!("⚠️  Warning: Shallow clone detected. Git history is incomplete.");
    eprintln!("   Run: git fetch --unshallow");
}
```

#### 2.3 "Still Present" Computation
**Decision:** Invert the operation (scan HEAD first, then check historical findings against it)

**Rationale:** O(1) lookup per finding vs O(n) tree search per finding

```rust
// Phase 1: Scan HEAD
let head_secrets: HashSet<SecretHash> = scan_tree(head_tree)?;

// Phase 2: Walk history
for finding in walk_history(repo)? {
    finding.still_present = head_secrets.contains(&finding.secret_hash);
}
```

#### 2.4 Memory Management
**Decision:** 
- Skip binary files (null byte detection in first 8KB)
- Per-blob size limit: 1MB default, configurable via `--max-file-size`
- Stream diff hunks, not full files

**Potential Issues:**
- Large repos (Linux kernel: 1M+ commits) need incremental scanning
- Need `--commits` limit and `--incremental` flag with watermark file

### 3. Credential Verification Framework

**Decision:** Defer full implementation to v0.9.0, but design the trait now

**Rationale:** 
- High surface area (many providers)
- Security-critical (must get error handling, rate limiting right)
- Not required for git history MVP

**Trait Design (for future):**

```rust
pub enum VerificationResult {
    Active { identity: Option<String>, scopes: Vec<String> },
    Inactive,
    Inconclusive { reason: String },
    NotAttempted { reason: String },
}

pub trait SecretVerifier: Send + Sync {
    fn handles(&self) -> &[&str];  // Rule IDs this verifier handles
    async fn verify(&self, secret: &str) -> Result<VerificationResult>;
}
```

**Security Requirements (from V82DEEPDIVE.md):**
- Never log full secret values
- Sanitize HTTP errors (may contain secrets in URLs/headers)
- Rate limiting per provider (token bucket: 5 req/sec)
- Deduplication before verification (same secret in 50 commits = 1 verification)

### 4. Entropy Detection Hardening

**Decision:** Implement comprehensive exclude lists and tunable thresholds

**Configuration:**

```rust
pub struct EntropyConfig {
    pub hex_threshold: f64,        // Default: 4.5
    pub base64_threshold: f64,     // Default: 4.0
    pub min_length: usize,         // Default: 20
    pub exclude_uuids: bool,       // Default: true
    pub exclude_css_colors: bool,  // Default: true
    pub exclude_lock_files: bool,  // Default: true
    pub exclude_minified: bool,    // Default: true
}
```

**Exclude Patterns:**
- UUIDs: `[0-9a-f]{8}-[0-9a-f]{4}-...`
- CSS colors: `#[0-9a-f]{3,8}`
- Lock file hashes: `package-lock.json`, `yarn.lock`, `Cargo.lock`, `go.sum`
- SRI hashes: `integrity="sha384-..."`
- Minified files: `*.min.js`, `*.min.css`

### 5. Benchmark Suite

**Decision:** Build honest, multi-dimensional benchmarks (not cherry-picked)

**Structure:**

```
coax-benchmarks/
├── datasets/
│   ├── secrets/
│   │   ├── true-positives/
│   │   ├── true-negatives/
│   │   └── encoded/
│   ├── unicode/
│   │   ├── bidi-attacks/
│   │   ├── homoglyphs/
│   │   ├── invisible-chars/
│   │   └── safe-i18n/
│   └── git-history/
├── run-benchmarks.sh
└── results/latest.md
```

**Metrics:**
- True Positive Rate (Recall)
- False Positive Rate
- Precision
- F1 Score
- Scan Speed (files/sec)
- Unicode Coverage (categories detected)

**Key Principle:** Include held-out test set that we never optimize against.
